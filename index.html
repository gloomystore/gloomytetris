이벤트 루프 

                    nodejs 혹은 브라우저엔진(크로뮴)이 담당
                    
                    자바스크립트는 싱글스레드 언어라서 모든 작업을 동기적으로 진행하게 되면, 로딩이 걸리는 작업은 다른 모든 스크립트를 멈추게 한다. 이를 해결하기 위해 이벤트루프라는 설계를 사용하게 되었다. 자바스크립트가 실행되면 stack에 실행컨텍스트가 생성되며, 실행 컨텍스트 생성 시 변수나 함수의 호이스팅이 일어난다. 함수의 실행이 완료되면 실행컨텍스트는 사라지며, 이 때 큐에 있던 작업 내용이 스택에 들어가게 된다.
                    
                    Queue에는 ajax나 settimeout과 같은 비동기 작업이 들어가게 되는데, 위 API들은 크로뮴엔진의 WEB APIs라고 부르며, 해당 비동기 작업이 끝나게 되면 해당 API들의 콜백함수가 콜백큐에 들어가게 된다.
                    이렇게 queue에 들어간 콜백함수는 stack에 비어있을 시 stack으로 진입하여 함수를 실행하게 된다.

Nextjs 

                      서버 측에서 페이지의 미리 렌더링을 수행하여 사용자가 페이지를 로드할 때 초기 로딩 시간을 줄인다.
                       
                      검색엔진 최적화에도 도움이 되며, 기존 PHP의 SSR 방식과 같이 서버사이드에서 이미 렌더가 된 페이지를 전송해주기 때문에 클라이언트에서 렌더할 내용이 줄어들어 로딩속도 향상에 기여할 수 있다. 
                      
                      Nextjs에서는 SSG, SSR 두가지 방식을 사용하고 있다.
                    

                    [SSG]
                    - getStaticProps 
                    빌드시에만 딱 한 번 fetch하여 캐시로 만들어 둔다. 
                    SSR과의 차이점은, SSR의 경우 클라이언트의 요청시 매번 렌더를 해서 HTML 내용을 만들어내는 반면, SSG는 빌드시에 딱 한번 렌더해서 내용을 미리 만들어둠으로써 서버의 자원소모를 줄인다. 

                    [SSR]
                    - getServerSideProps
                    기존의 SSR방식과 유사하며, JSX serverSideProps를 바인딩 해두는 것만으로도 서버에서 미리 렌더해서 내용을 보여줄 수 있다. 요청받는 매번 getServerSideProps 내부의 함수를 실행하기 때문에 기존의 SSR과 같이 서버에서 렌더하는 연산 비용이 들게된다. 
                    
                    [서버 컴포넌트]
                    js파일 안에서 소문자로 컴포넌트를 만들면 서버컴포넌트가 된다. getServerSideProps와 동일한 기능
                  	

                      - 더이상 link 태그 안에 a태그를 넣지않음
                      - client 컴포넌트와 서버 컴포넌트의 분리
                      - 기본적으로 default가 서버 컴포넌트이며, 컴포넌트 이름 앞이 대문자가 아니면 모두 서버컴포넌트 취급된다.

Vue 

[beforeCreate]
                      인스턴스가 생성되고 리액티브한 데이터가 준비되기 전에 호출됩니다. 이 훅에서는 data, computed, watch 등의 옵션에 접근할 수 없습니다. 대신에 인스턴스를 초기화하는 코드를 작성할 수 있습니다.

                      [created]
                      인스턴스가 생성된 후에 호출됩니다. 이 훅에서는 data, computed, watch 등의 옵션에 접근할 수 있습니다.

                      [beforeMount]
                      인스턴스가 마운트되기 전에 호출됩니다. 이 훅에서는 템플릿이 렌더링되기 전에 DOM에 직접적인 접근이 가능합니다.

                      [mounted]
                      컴포넌트가 DOM에 마운트된 직후에 호출됩니다. 이 때 DOM에 접근하여 필요한 작업을 수행할 수 있습니다. 예를 들어, 외부 라이브러리를 로드하고 초기화하는 작업 등을 수행할 수 있습니다.

                      [beforeUpdate]
                      컴포넌트가 업데이트되기 직전에 호출됩니다. 이 훅은 컴포넌트의 데이터나 prop 등이 변경되었을 때 실행됩니다. beforeUpdate 훅에서 데이터를 변경하는 것은 좋지 않으며, 데이터 변경 시점은 updated 훅에서 처리하는 것이 좋습니다.
                      
                      [updated]
                      컴포넌트가 업데이트된 직후에 호출됩니다. 이 때 DOM이 업데이트되었으므로, 업데이트된 데이터에 대한 처리를 수행할 수 있습니다.

                      [beforeUnmount]
                      컴포넌트가 제거되기 직전에 호출됩니다. 이 때 컴포넌트에 등록한 이벤트 리스너나 타이머 등을 제거하는 등의 작업을 수행할 수 있습니다.

                      [unmounted]
                      컴포넌트가 DOM에서 제거될 때 실행됩니다. 이 훅은 주로 컴포넌트가 사용하는 외부 리소스를 정리하고 메모리 누수를 방지하는 데 사용됩니다.
                      컴포넌트 인스턴스가 파괴된 후 실행되기 때문에 컴포넌트 인스턴스에 접근할 수 없습니다. 따라서 unmounted 훅에서는 인스턴스 내부의 데이터, 메서드, 프로퍼티 등에 접근할 수 없습니다.
                      
                      [computed]
                      특정 값의 계산 결과를 캐싱하여 저장하고, 값이 변경될 때만 새로 계산합니다. 즉, 불필요한 계산을 줄이기 때문에 성능상 이점이 있습니다. computed는 데이터에 의존적인 값을 계산하는 데 사용됩니다.
                      
                      [watch]
                      watch는 데이터의 변화를 감지하고, 변화에 따라 특정 동작을 수행합니다. watch는 데이터의 값을 감시하고 있으며, 값이 변경될 때마다 특정 동작을 실행합니다. watch는 데이터의 값을 감시하는 데 사용됩니다.

웹팩 

1. Entry
                      Entry는 웹팩이 빌드할 파일의 진입점을 지정하는 부분입니다. Entry를 설정하여 해당 파일을 시작으로 의존성을 찾아 모든 파일을 묶어내게 됩니다. Entry는 다음과 같은 형태로 설정할 수 있습니다.
                      
                      ./src/index.js 파일이 웹팩 빌드의 진입점으로 지정되었습니다.
                      

module.exports = {
  entry: './src/index.js',
  // ...
};

                   
                    2. Output
                    Output은 웹팩이 빌드한 파일의 출력 경로와 파일 이름을 지정하는 부분입니다. Output은 다음과 같은 형태로 설정할 수 있습니다.
                    
                    path는 출력 파일 경로를 지정하는 부분이고, filename은 출력 파일 이름을 지정하는 부분입니다.
                    

const path = require('path');

module.exports = {
  // ...
  output: {
    path: path.resolve(__dirname, 'dist'),
    filename: 'bundle.js',
  },
};

                   
                    3. Loader
                    Loader는 웹팩이 빌드할 때 각 파일을 처리하는 방법을 지정하는 부분입니다. 주로 CSS, 이미지, 폰트 등의 파일을 로드하고, JavaScript로 변환하여 웹팩이 이해할 수 있는 형태로 변환합니다. Loader는 다음과 같은 형태로 설정할 수 있습니다.
                    
                    test는 Loader가 처리할 파일의 확장자를 지정하는 부분이고, use는 Loader가 실제로 사용될 때 어떤 Loader를 사용할지 지정하는 부분입니다.
                   

module.exports = {
  // ...
  module: {
    rules: [
      {
        test: /\.css$/,
        use: ['style-loader', 'css-loader'],
      },
    ],
  },
};

                   
                   
                    4. Plugin
                  Plugin은 웹팩의 빌드 과정 중에 추가적인 작업을 수행하는 부분입니다. 주로 파일 압축, 번들링 결과의 HTML 파일 생성, 환경 변수 주입 등의 작업을 수행합니다. Plugin은 다음과 같은 형태로 설정할 수 있습니다.
                  
                  HtmlWebpackPlugin은 번들링 결과를 HTML 파일로 생성하는 플러그인입니다. template은 HTML 템플릿 파일을 지정하는 부분이고, filename은 생성될 HTML 파일의 이름을 지정하는 부분입니다.
                   

const HtmlWebpackPlugin = require('html-webpack-plugin');

module.exports = {
  // ...
  plugins: [
    new HtmlWebpackPlugin({
      template: './src/index.html',
      filename: 'index.html',
      hash: true,
    }),
  ],
};

                  
                  5. Mode
                  웹팩의 Mode는 개발 환경 또는 프로덕션 환경에서 어떤 옵션을 적용할지 설정하는 속성입니다.
                  개발 환경에서는 Mode: 'development'를 설정하고, 프로덕션 환경에서는 Mode: 'production'을 설정합니다.
                  개발 환경에서는 빠른 빌드를 위해 디버그 정보가 포함된 개발용 코드가 출력되며, 프로덕션 환경에서는 최적화된 코드가 출력됩니다.
                  예시:
                   

// 개발용 웹팩 설정 파일
module.exports = {
  mode: 'development',
  // ...
}

// 프로덕션용 웹팩 설정 파일
module.exports = {
  mode: 'production',
  // ...
}

                    

웹팩은 SPA와 사용될 때, 실시간 번들링으로 수정하는 코드가 실시간으로 화면에 반영되도록 한다.
                    
                    빌드시에는 번들링할 때 import된 모듈만 번들링하며, 동시에 minify되기 때문에 용량, 속도면에서 이익이 있다.
                    
                    이외에 css파일을 모듈화해서 가져오는 등 부가적인 기능이 있다.

계층 구조
						PDU
                    

                    3way
                    1. 클라이언트 -> 서버로 SYN패킷 전송 -> 이때 sequence의 값을 x로 보낸다.
                    2. 서버가 SYN패킷을 받고, 클라이언트에게 확인 신호인 ACK(x+1) 패킷을 보낸다. -> 이때 sequence의 값을 y로 보낸다.
                    3. 클라이언트는 서버의 응답 ACK(x+1)과 SYN(y)를 받은 뒤에, 확인신호로 ACK(y+1)을 보낸다
                    
                    4way
                    1. 연결 해제를 위해 FIN 패킷 전송
                    2. 서버는 FIN패킷을 받고 CLOSE_WAIT 상태를 유지하며 ACK 패킷을 보냄
                    3. 서버는 데이터를 모두 보낸 뒤에 FIN패킷을 보내 연결이 해제 되었음을 알림
                    4. 클리이언튼 FIN패킷을 받고, TIME_WAIT 상태를 유지하다가 데이터를 다 받으면, ACK를 서버에 전송
                    -> TIME_WAIT 시간이 끝나면 클라이언트는 소켓을 닫음

                   [HTTP/1.1]
                   keep-alive 지원: 웹사이트에서 보내는 resource(이미지, css, html 등)을 매번 요청을 받을 때마다 따로 보내주지 않고, 클라이언트의 정보를 저장해놓고 (나중에 찾아봄)
                    

HTTP 기본 메소드(GET,POST,PUT,DELETE등)을 사용하지 않고도 개발은 가능하다.
                    하지만 약속을 해두지 않으면 각자 개발방식에 맞추기 힘들기 때문에 REST라는 규약을 지어둔다. (Representational State Transfer)
                    
                    [REST]
                    - 자원(resource): URI를 이용
                    - 행위(verb): HTTP 메소드를 이용
                    - 표현(representational): payload를 이용
                    

소셜로그인에서 주로 사용되며, 로그인 정보가 아니라 (로그인 정보는 없음!)
                    이 사용자가 [resource server(구글 등)]에 존재하는 사용자이며, 필요한 해당 사용자의 정보만을 [Access token]에 담아 인증서버에 보내준다. 
                    그리고 [인증 서버]는 [클라이언트]에 그 정보를 보내준다. (화면에서 자신의 이메일이나 이름 등 확인 가능)
                    

                    Header.Payload.Signature 가 해쉬된 형태로 클라이언트에 전달
                    
                    header: alg:sha256 typ: jwt
                    payload: id:admin
                    signature:  위 두 내용에 secrey key를 조합

CS 

                      1. 우선 ISP의 DNS를 통해 구글 도메인이 갖는 아이피를 반환 받는다.
                      2. 반환받은 아이피로 GET요청을 보내게 된다.
                      3. nginx와 리버스프록시를 사용한다고 가정하고, 아마도 앵귤러 프로젝트가 실행되고 있는 포트로 연결될 것
                      4. 앵귤러 프로젝트의 번들링된 빌드 파일들이 response되고, 내 브라우저에서 해당 파일들의 다운로드와 로딩이 끝나면 화면을 볼 수 있게 된다.
                      5. 다만 구글의 경우 메인화면은 SSR을 사용하고 있으므로, js가 로딩되기 전에 모든 화면이 보일 것.
                    

                      숫자로된 아이피는 사람들이 외우기 어렵기 때문에, 도메인이라는 시스템이 도입 되었다.
                      
                      DNS는 Domain Name System의 준말로, 받은 도메인을 아이피로 반환하여 준다.
                      
                      처음에는 ISP에서 제공하는 로컬 DNS에서 찾다가, 없으면 점점 root DNS(전세계에 13개 존재하는)쪽으로 찾게 된다.
                      
                      그렇게 ip를 찾게되면 해당 아이피를 가지고 있는 서버로 브라우저의 get요청이 향하게 된다.
                    

                      get은 데이터를 가져오는 요청이며, url에 파라미터를 붙이기 때문에 만약 파라미터를 붙일 일이 있으면 보안과 상관없는 내용이 붙어야한다. 주로 검색에 사용된다.
                      
                      post는 서버로 데이터를 보내는 요청이며, body가 url에 드러나지 않기 때문에 더 보안에 강하다. 주로 로그인, 회원가입에 사용된다.
                    

                      자원을 이름(자원의 표현)으로 구분하여 해당 자원의 상태(정보)를 주고 받는 모든 것을 의미한다.
                      
                      특징: URL로 직관적으로 어떤 resource를 요구하는지 알 수 있다.
                      
                      구성요소:
                      자원(Resource) - URI(Uniform Resource Identifier)를 통해 자원을 명시하고 해당 자원의 상태(정보)를 주고받습니다. 예를 들어, /users라는 URI는 모든 사용자의 목록을 가져오는 자원을 나타냅니다.

                      행위(Verb) - HTTP Method(GET, POST, PUT, DELETE)를 사용하여 해당 자원에 대한 행위를 정의합니다. 예를 들어, GET 메서드는 해당 자원의 상태를 가져오는 데 사용되며, POST 메서드는 해당 자원에 새로운 자원을 추가하는 데 사용됩니다.

                      표현(Representation) - 클라이언트와 서버 간의 통신에서 사용되는 데이터 표현 방식입니다. 일반적으로 XML이나 JSON 형태의 데이터를 사용합니다.
